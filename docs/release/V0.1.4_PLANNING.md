## Release Theme

Enterprise Hardening and Operator Experience

v0.1.4 should improve operator confidence, local diagnostics, release discipline, installer reliability, and policy enforcement while preserving the local-first and safety-first design of ZAI Coder.

## Proposed Workstreams

### 1. CLI Operator UX and Diagnostics

Goals:
- Make command output clearer and more consistent.
- Improve failure messages for local operators.
- Ensure `--help`, `--version`, and subcommand help remain stable.
- Add diagnostics for environment, config, and runtime paths.

Candidate tasks:
- Add `zai-coder doctor` or improve existing diagnostics command if present.
- Add tests for common CLI error paths.
- Add help-output snapshot or semantic tests for important commands.
- Document CLI troubleshooting examples.

### 2. Policy and Safe Runner Hardening

Goals:
- Expand protection for dangerous shell patterns.
- Keep safety policy explicit, testable, and auditable.
- Improve messages when commands are blocked.

Candidate tasks:
- Add regression tests for additional dangerous git patterns.
- Add tests for shell chaining and indirect destructive commands.
- Add docs for allowed vs blocked command examples.
- Verify blocked command messages are useful.

### 3. Installer / Update / Uninstall Reliability

Goals:
- Make local install and uninstall safer.
- Improve post-install diagnostics.
- Ensure failure states are understandable.

Candidate tasks:
- Add tests for idempotent install/uninstall behavior.
- Verify post-install check output.
- Add docs for local install recovery.
- Improve path handling and shell portability.

### 4. Test Coverage and CI Reliability

Goals:
- Keep test suite deterministic and local-first.
- Prevent workflow regressions.
- Ensure repo checks stay fast and meaningful.

Candidate tasks:
- Add tests for CI guard scripts.
- Add tests for repo-check script failure modes.
- Add a lightweight validation matrix doc.
- Ensure no generated DB/evidence/output files return to the index.

### 5. Release Automation Guardrail Improvements

Goals:
- Make future release publish phases harder to misuse.
- Improve release checklist evidence.
- Preserve signed tag and asset auditability.

Candidate tasks:
- Add script to print release readiness status.
- Add dry-run validation for tag/release preconditions.
- Add tests for release checklist invariants.
- Add docs for emergency recovery and no-history-rewrite policy.

### 6. Documentation and Examples Polish

Goals:
- Improve onboarding and operator confidence.
- Clarify local-first architecture and safety model.
- Keep examples aligned with guarded release process.

Candidate tasks:
- Update README release/install sections if needed.
- Add docs index for ops and release docs.
- Add short examples for CLI usage.
- Add troubleshooting docs for CI/dependency setup.

### 7. Optional Server/API Diagnostics

Goals:
- Improve local server observability if server mode is actively supported.
- Keep API diagnostics local-only and testable.

Candidate tasks:
- Add tests for health/status endpoints.
- Document local server startup/shutdown.
- Add diagnostics around ports and config.

## Release Gates for v0.1.4

Before release candidate:
- Full tests pass.
- repo-check passes.
- secret-scan passes.
- stage-manifest-check passes.
- generated-state guard passes.
- CI pytest setup guard passes.
- package-check passes when package artifacts are generated locally.
- release notes and checklist are complete.

Before publish:
- release candidate PR merged.
- main synced.
- version verified on main.
- clean working tree.
- signed tag created only after main verification.
- GitHub release created only after artifacts and checksums pass.
